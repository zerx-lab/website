## 🔥 今日焦点

### GitHub 正式发布 Claude 和 Codex AI 代理公开预览 ⭐⭐⭐⭐⭐

**核心要点：**
- GitHub 于 2026 年 2 月 4 日宣布 Claude 和 Codex 作为编码代理正式进入公开预览
- Copilot Pro+ 和 Copilot Enterprise 用户可在 github.com、GitHub Mobile 和 VS Code 中直接使用
- 用户可从 issues、pull requests、仓库 Agents 标签以及 VS Code 代理会话视图启动代理会话
- Claude 代理支持已集成到 GitHub Copilot VS Code 扩展 v1.109（2026 年 1 月发布）

**技术解读：**

GitHub 将 AI 代理的概念从简单的代码补全工具升级为可以自主完成复杂任务的编程伙伴。与传统的代码建议不同，AI 代理可以：

1. **理解并执行完整的开发任务**：从 issue 描述直接生成 PR，而不仅仅是建议几行代码
2. **跨平台无缝协作**：在 github.com、移动端和 IDE 之间保持会话连续性
3. **工具驱动交互**：通过 Model Context Protocol (MCP) 集成，代理可以调用外部工具、API 和服务
4. **多模型选择**：开发者可以根据任务特点选择 Claude（擅长推理和长文本）或 Codex（针对编程优化）

GitHub Copilot v1.109 引入的关键特性包括：
- **Claude Agent SDK 集成**：直接使用 Anthropic 官方 SDK，确保最佳性能
- **丰富的 MCP 应用交互**：在 VS Code 内实现更工具驱动、交互式的 Copilot 体验
- **改进的代理会话管理**：跨设备同步代理工作状态

这标志着 GitHub 从"AI 辅助编程"迈向"AI 协作编程"的关键转折点。Copilot 不再只是提供建议，而是成为可以委派任务的团队成员。

**背景上下文：**

自 2021 年 GitHub Copilot 推出以来，AI 辅助编程已经从新奇技术转变为开发者的日常工具。但直到 2026 年，大多数 AI 编程助手仍然局限于"补全下一行代码"的范式。

2026 年初，AI 代理（Agentic AI）成为行业焦点。与被动响应不同，代理可以主动规划、执行多步骤任务、调用工具并自我验证结果。GitHub 此次整合 Claude 和 Codex 代理，是将这一范式应用于全球最大的代码托管平台。

值得注意的是，GitHub 选择了多模型策略而非独家绑定 OpenAI：
- **Claude**：擅长复杂推理、长上下文理解和自然语言交互
- **Codex**：OpenAI 专为编程优化的模型，理解代码结构和 API

这种开放性反映了开发者需求的多样性——不同任务需要不同的 AI 能力。

**开发者行动建议：**
- **立即尝试**：如果你是 Copilot Pro+ 或 Enterprise 用户，访问 github.com 或 VS Code 开始使用代理功能
- **从小任务开始**：尝试让代理处理简单的 bug 修复或文档更新，逐步建立信任
- **选择合适的模型**：复杂架构设计用 Claude，快速代码生成用 Codex
- **利用跨平台特性**：在移动端查看 issue 时启动代理，回到桌面继续同一会话
- **探索 MCP 集成**：如果你的工作流依赖特定工具，研究如何通过 MCP 让代理访问它们
- **提供反馈**：公开预览阶段是影响产品方向的最佳时机

### MIT 递归语言模型突破 1000 万 Token 上下文窗口限制 ⭐⭐⭐⭐⭐

**核心要点：**
- MIT CSAIL 研究团队发布递归语言模型（RLM），可处理超过 1000 万 token 的输入
- 在 BrowseComp+ 任务（1K 文档，600-1100 万 token）上达到 91.33% 准确率
- 相比传统方法性能提升高达两个数量级，同时成本降低 3 倍
- 无需重新训练模型，可在现有 LLM（如 GPT-4、Claude）上直接应用

**技术解读：**

大语言模型的上下文窗口一直是限制其能力的关键瓶颈。虽然最新的模型已经支持 100 万甚至 200 万 token，但当输入长度达到数百万 token 时，传统方法会遭遇"上下文腐烂"（context rot）——模型难以在海量信息中找到相关细节，性能急剧下降。

MIT 的递归语言模型（RLM）通过一个巧妙的范式转变解决了这个问题：**将输入视为外部环境而非内存的一部分**。

具体实现方式：
1. **Python REPL 笔记本环境**：完整的输入文档被加载到一个 Python 变量中
2. **符号化交互**：模型不直接"记住"所有内容，而是编写代码来与文档交互
3. **递归查询**：模型可以执行 `grep`、分区、递归子查询等操作来定位相关信息
4. **选择性上下文加载**：只将当前任务所需的信息片段加载到上下文窗口

以处理 1000 万 token 的法律文档库为例：
- 传统方法：尝试将所有文档塞进上下文窗口（失败或性能极差）
- RLM 方法：模型先写代码检索相关案例，然后仅对这些案例进行深度分析

这种方法的优势在于：
- **突破硬件限制**：即使模型训练时只支持 27.2 万 token，也能处理 1000 万 token 输入
- **性能提升**：在长文档任务中，RLM 使用 GPT-5 达到 91.33% 准确率，远超基础模型和摘要方法
- **成本优化**：选择性加载上下文使 API 调用成本降低 3 倍

**背景上下文：**

上下文窗口的扩展一直是 LLM 发展的主要竞争赛道。2023 年初，GPT-4 的 32K 上下文已是行业领先；到 2025 年底，Gemini 和 Claude 已经突破 100 万 token。但这种"暴力扩展"面临两大挑战：

1. **计算成本呈平方增长**：Transformer 的注意力机制复杂度为 O(n²)，100 万 token 的成本是 10 万 token 的 100 倍
2. **上下文腐烂**：模型在超长输入中"迷失"，无法准确定位关键信息

RLM 的创新之处在于不依赖更大的上下文窗口，而是让模型"更聪明地使用有限的窗口"。这类似于人类阅读：我们不会一次性记住整本书，而是根据需要翻阅相关章节。

**开发者行动建议：**
- **关注 GitHub 仓库**：RLM 的代码和论文已公开，可以尝试在你的长文档任务中应用
- **重新审视长上下文需求**：如果你的应用需要处理大量文档，考虑 RLM 而非依赖更大的上下文窗口
- **实验递归策略**：尝试让 LLM 先检索再分析，而非一次性处理所有内容
- **优化成本**：对于需要处理长文档的应用，RLM 可能大幅降低 API 成本
- **关注后续发展**：RLM 可能成为 2026 年 LLM 推理的重要范式

### APT28 利用 Microsoft Office 零日漏洞 CVE-2026-21509 发动攻击 ⭐⭐⭐⭐⭐

**核心要点：**
- 俄罗斯 APT28 黑客组织在 Microsoft 公开漏洞 3 天后即开始利用 CVE-2026-21509
- 该零日漏洞（CVSS 评分 7.8）允许攻击者通过恶意 RTF 文档绕过 Office 安全功能
- 攻击目标包括乌克兰、斯洛伐克和罗马尼亚的政府机构，已有超过 60 个电子邮件地址受到攻击
- 部署的恶意软件包括 MiniDoor（窃取电子邮件）和 Covenant Grunt（高级后门）
- CISA 已将该漏洞加入 KEV 目录，要求联邦机构在 2026 年 2 月 16 日前修补

**技术解读：**

CVE-2026-21509 是 Microsoft Office 中的安全功能绕过漏洞，利用了 Office 处理 RTF（富文本格式）文件的缺陷。与传统的 Office 漏洞不同，该漏洞**无需启用宏或用户交互即可触发**——只要打开恶意 RTF 文档，攻击代码就会自动执行。

攻击链如下：
1. **鱼叉式钓鱼邮件**：APT28 发送包含地缘政治主题的邮件（如跨国武器走私、军事训练计划）
2. **恶意 RTF 文档**：邮件附件是精心构造的 RTF 文件，利用 CVE-2026-21509
3. **代码执行**：打开文档时，漏洞触发并下载第一阶段 payload
4. **恶意软件部署**：
   - **MiniDoor**：轻量级后门，专门窃取电子邮件和文档
   - **PixyNetLoader**：加载器，用于部署更复杂的 Covenant Grunt 后门
   - **Covenant Grunt**：高级 C2 框架，可执行任意命令、窃取凭证、横向移动

APT28 的快速响应能力令人震惊：
- **1月26日**：Microsoft 公开披露 CVE-2026-21509
- **1月27日**：文件元数据显示恶意文档已创建
- **1月29日**：Zscaler ThreatLabz 观察到攻击活动

这表明 APT28 可能在 Microsoft 披露前就已经了解该漏洞，或者拥有极快的漏洞武器化能力。

**背景上下文：**

APT28（又名 Fancy Bear、Sofacy）是俄罗斯军事情报机构 GRU 关联的高级持续性威胁组织，活跃于全球网络间谍活动。该组织以快速利用零日漏洞、针对地缘政治目标发动攻击而闻名。

此次攻击是"Operation Neusploit"行动的一部分，主要针对：
- 乌克兰中央行政机构（超过 60 个电子邮件地址）
- 斯洛伐克和罗马尼亚的政府部门
- 与军事、安全相关的组织

CVE-2026-21509 的严重性在于：
1. **无需用户交互**：传统 Office 漏洞通常需要启用宏，但此漏洞在打开文档时自动触发
2. **广泛影响**：Office 2016 和 2019 用户易受攻击（Office 2021 受保护）
3. **政治动机**：APT28 专注于地缘政治情报收集，目标高度精准

CISA 将该漏洞添加到已知被利用漏洞（KEV）目录，反映了其在野外被积极利用的严重威胁。

**开发者行动建议：**
- **立即修补**：如果使用 Office 2016/2019，立即安装 Microsoft 于 1 月 26 日发布的紧急补丁
- **审计电子邮件**：检查 1 月 26 日后收到的带有 RTF 附件的邮件，特别是地缘政治主题
- **网络隔离**：将关键系统与互联网隔离，限制恶意软件横向移动
- **监控异常**：检查日志中是否有可疑的外部连接、凭证访问或文件窃取活动
- **更新威胁情报**：订阅 CERT-UA、CISA 等机构的安全警报
- **考虑升级**：如果可能，迁移到 Office 2021 或 Microsoft 365，它们默认受保护
- **加强安全意识**：培训员工识别钓鱼邮件，特别是地缘政治或军事主题的邮件

### Vue 3.6 Vapor Mode 和 Vite+ 统一工具链 ⭐⭐⭐⭐
