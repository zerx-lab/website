### LinkedIn 身份验证深度调查：三分钟验证，你究竟交出了什么？ ⭐⭐⭐⭐⭐

**核心要点：**
- 一篇深度调查《I Verified My LinkedIn Identity. Here's What I Handed Over》在 Hacker News 斩获今日最高分 **1,091 分、400+ 条评论**
- LinkedIn 身份验证通过第三方服务商 **Persona**（国际版）或 CLEAR（北美版）完成，仅三分钟验证即采集：**面部几何生物特征**（从自拍和护照提取的数学面部映射）、**护照 NFC 芯片数据**、行为生物特征
- 关键风险：Persona 是加州注册公司，受美国 **CLOUD Act** 约束——即便数据存储在美国境外，美国执法机构仍可强制要求其提供数据
- 数据留存漏洞：LinkedIn 声称 14 天内永久删除，但 Persona 的政策设有例外条款——"法律要求留存"可无限期保存
- 生物特征数据与密码不同，**面部几何数据一旦泄露无法更改**

**技术解读：**

这不是普通的"隐私条款"问题，而是针对职业社交平台身份验证架构的深度风险评估。LinkedIn 的验证流程将数据主权实际转交给第三方处理商，用户对自己的生物特征数据去向几乎没有控制权。更关键的是，"零知识"验证技术（如 ZK Proof）在技术上已可实现身份证明而无需存储原始生物特征，但 LinkedIn 选择了更简便、更便宜的传统方案。

对于欧盟用户，GDPR 提供了额外保护——Persona 须在 30 天内响应删除请求。但对于非欧盟用户，数据控制权极为有限。

**背景上下文：**

LinkedIn 于 2022 年引入身份验证徽章功能，旨在打击虚假账户。但随着功能普及，大量用户在不了解数据收集范围的情况下完成了验证。这篇调查的爆红（1,091 分）说明开发者社区对此高度敏感。

**开发者行动建议：**
- 如已完成 LinkedIn 身份验证，发邮件至 **idv-privacy@withpersona.com** 请求数据副本和删除
- 欧盟用户可依据 GDPR 第 17 条（被遗忘权）要求删除，Persona 须在 30 天内响应
- 企业如正在构建身份验证系统，评估 ZK Proof 或零知识身份方案，避免存储原始生物特征

### Karpathy："Claws" 正在成为 LLM Agent 之上的新架构层 ⭐⭐⭐⭐⭐

**核心要点：**
- Andrej Karpathy 发布推文，正式提出 **"Claws"** 概念：就像 LLM Agent 是 LLM 之上的新层，Claws 是 LLM Agent 之上的新层，将编排、调度、上下文、工具调用和持久化提升到新水平
- Hacker News 标题《Claws are now a new layer on top of LLM agents》获 **150 分**
- Claws 运行在个人设备上，通过消息协议通信，既可响应直接指令，也可自主调度任务——相当于"有记忆、有日程、能主动行动的 Agent 框架"
- 当前出现多个实现：`OpenClaw`（大型，安全争议），`NanoClaw`（~4,000 行代码，可审计），另有 `zeroclaw`、`ironclaw`、`picoclaw` 等正在涌现
- Karpathy 对 `OpenClaw` 持审慎态度：代码规模庞大，正在被大规模攻击，已出现 RCE 漏洞、供应链污染和恶意插件报告；倾向于使用代码量在脑容量内的 `NanoClaw`

**技术解读：**

Claws 的核心创新在于将"代理"的边界从单次任务延伸到跨会话、跨时间的持久化自主行为。传统 LLM Agent 是无状态的任务执行器；Claws 增加了长期记忆、主动调度和工具调用的持久化层，使 Agent 可以"记住上周做了什么"并"明天主动触发某个任务"。

从架构演进看：**LLM → LLM Agent（单次任务）→ Claws（持久化、自主调度）**。这与操作系统的进程管理模型有相似之处，Claws 相当于为 Agent 引入了进程调度器和持久化文件系统。

**开发者行动建议：**
- 关注 `NanoClaw` 等小型实现，优先选择代码量可审计的方案
- 在生产环境部署 Claws 时，所有操作建议在容器内隔离运行
- 参考 Karpathy 的安全建议：给私有数据/密钥提供访问权限前，务必审计代码来源

### Cloudflare 2月20日大规模故障事后分析：空参数酿成 6 小时全球宕机 ⭐⭐⭐⭐⭐

**核心要点：**
- Cloudflare 官方发布 2026 年 2 月 20 日故障事后分析，故障持续 **6 小时 7 分钟**（UTC 17:48 - 23:55）
- 根本原因：自动化清理任务的 API 查询逻辑缺陷——将 `pending_delete` 参数**空值**误解释为"所有前缀"，而非"无匹配前缀"，触发批量删除
- 影响范围：**25% 的 BYOIP 前缀**（4,306 个中的 1,100 个）被删除，核心 CDN 与安全服务、Spectrum、Magic Transit、专用出口服务全部受影响；1.1.1.1 网站返回 HTTP 403 错误（DNS 解析本身持续正常）
- 恢复难点：大量时间花费在逐一恢复已删除配置和服务绑定，而非诊断根因

**技术解读：**

这是一个经典的"空值边界条件"缺陷——API 设计中将"未提供参数"等同于"返回全部"是一种常见但危险的模式，在批量操作中会引发灾难性后果。正确的设计应为：无参数时返回空集或报错，而非默认匹配所有。

此次故障还暴露了自动化运维脚本的权限过度问题：一个清理任务能在无二次确认的情况下批量删除生产前缀配置。

**开发者行动建议：**
- 在 API 设计中，批量操作的默认行为应为"无匹配"而非"匹配全部"
- 自动化脚本执行破坏性操作前，建议增加 dry-run 模式和数量上限检查
- 基础设施配置变更应实现"先备份后执行"，确保可回滚

### ETH Zurich 研究：Bitwarden、LastPass、Dashlane 存在 25 项安全漏洞 ⭐⭐⭐⭐⭐

来自苏黎世联邦理工学院（ETH Zurich）与意大利卢加诺大学（USI）的研究团队发布论文，将于 USENIX Security 2026 正式发表。研究对 Bitwarden、LastPass、Dashlane 三大主流密码管理器（合计超过 6,000 万用户）发现共 **25 项安全攻击**：Bitwarden 12 项（其中 7 项可导致密码泄露）、LastPass 7 项（3 项密码泄露）、Dashlane 6 项（1 项密码泄露）。

攻击模型基于"恶意服务器"假设：若服务商服务器被攻破并开始恶意行为，攻击者仅需用户执行登录、打开保险库、查看密码等常规操作即可获取明文密码，**无需强力计算基础设施**。研究团队指出"零知识加密"并非密码学标准术语，而是营销用语，各家厂商对此定义不一。

例外：**1Password** 的架构因在密钥派生中引入高熵"Secret Key"，使暴力破解无法实现，被评为最安全。

**为什么重要：** 揭示了密码管理器"零知识"承诺的实际局限，供开发者和企业安全团队重新评估密码管理方案选型。
