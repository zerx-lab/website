### Chrome DevTools MCP：让 AI 编码助手直接调试你的浏览器会话 ⭐⭐⭐⭐⭐

**核心要点：**

- Chrome M144+ 正式引入 **Chrome DevTools MCP（Model Context Protocol）服务器**，使 AI 编码助手（如 Claude Code、Cursor 等）能够直接连接并操作用户的**实时浏览器会话**，包括已认证状态和正在进行中的调试面板
- 无需重新登录或创建新会话——Agent 可直接接管需要身份验证才能访问的页面进行调试，真正实现"接力式"人机协作排错
- 提供三种连接模式：专属 Profile、远程调试端口（`--remote-debugging-port`），以及新增的 `--autoConnect` 自动连接标志
- 每次连接均需用户通过 Chrome 弹窗**显式授权**，会话期间浏览器顶部显示"正由自动化测试软件控制"横幅
- 需在 `chrome://inspect/#remote-debugging` 开启远程调试功能

**技术解读：**

Chrome DevTools MCP 解决的是 AI Agent 调试中最核心的障碍：此前 Agent 只能在隔离环境中运行，无法接触需要登录或有特定用户状态的真实页面。现在开发者可以在排查问题到一半时，直接将 DevTools 中选中的网络请求或 DOM 元素交给 Agent 继续分析——这与 MCP 让 AI 访问外部工具的理念一脉相承，是浏览器层面的 MCP 落地。

**开发者行动建议：**

- 升级至 Chrome M144+，通过 `chrome://inspect/#remote-debugging` 启用远程调试
- 在 MCP 服务器配置中添加 `--autoConnect` 参数，体验无缝的 Agent 接管调试流程
- 注意权限边界：每次连接需手动批准，不要在公共或共享机器上启用此功能

### Glassworm 卷土重来：不可见 Unicode 攻击入侵 151 个 GitHub 仓库 ⭐⭐⭐⭐⭐

**核心要点：**

- 2026 年 3 月 3 日至 9 日，攻击组织 **Glassworm** 入侵至少 **151 个 GitHub 仓库**，受害者包括知名项目 Wasmer、Reworm（1,460 stars）及 anomalyco 的 opencode-bench
- 攻击武器是**私用区 Unicode 字符（PUA）**——这些字符在几乎所有编辑器、终端和代码审查界面中均渲染为空白或完全不可见，但实际携带可执行的恶意代码
- 隐藏代码被解码后传入 `eval()`，历史载荷曾通过 **Solana 链**上的投递通道窃取令牌、凭证和密钥
- 攻击面涵盖 **JavaScript/npm 包、GitHub 源码仓库、VS Code 插件市场**；Glassworm 使用 AI 生成高可信度的伪装提交信息，令人工审查几乎无效
- Glassworm 至少从 2025 年 3 月起持续活跃，本轮是迄今规模最大的一波攻击

**技术解读：**

这是供应链安全的噩梦场景：攻击向量对肉眼和传统 linter 完全不可见，且借助 AI 生成的合法提交掩护，代码评审形同虚设。与此前的 typosquatting 或恶意依赖不同，PUA Unicode 攻击直接污染受信任项目的源码，影响所有下游消费者。标准的 `npm audit` 和 `eslint` 均无法检测此类威胁。

**开发者行动建议：**

- 立即在 CI/CD 流水线中加入**非打印字符扫描**步骤（可使用 `grep -P '[\x{E000}-\x{F8FF}]'` 或专用工具）
- 审查近期合并的 PR，重点检查涉及字符串字面量的变更
- 对 VS Code 插件和 npm 包依赖树执行源码级校验，不要仅依赖哈希验证

### Go 1.26 `//go:fix inline`：源码级内联器一条命令完成百万级 API 迁移 ⭐⭐⭐⭐

**核心要点：**

- Go 1.26 在 `go fix` 子命令中引入**源码级内联器**：包作者在废弃函数上标注 `//go:fix inline`，下游用户执行 `go fix ./...` 即可自动将所有调用方内联迁移至新 API
- 典型用例：`ioutil.ReadFile` → `os.ReadFile`（函数重命名）、参数顺序修正（如 `Sub(y, x)` → `Sub(x, y)`）
- 内联器约 **7,000 行逻辑**，自动处理六大边界情况：副作用顺序、变量遮蔽、未使用变量、defer 语句、编译期常量错误、参数消除
- 已在 Google 内部 Monorepo 中准备超过 **18,000 个变更列表**，完成数百万次废弃 API 调用的自动迁移

**技术解读：**

`//go:fix inline` 将"API 废弃"的成本压缩到库作者一次性标注，类似于 Rust 的 `cargo fix` 在 Rust 2018 edition 迁移中扮演的角色，但直接集成进 Go 标准工具链。Go 生态的 API 演化门槛因此大幅降低。

**开发者行动建议：**

- 升级至 Go 1.26，运行 `go fix ./...` 批量迁移废弃 API
- 在维护的库中，考虑为即将废弃的函数添加 `//go:fix inline` 注解，为下游用户提供自动迁移路径

### C++26 正式纳入 Oxford Variadic Comma 标准 ⭐⭐⭐

C++26 委员会本周正式通过 **Oxford Variadic Comma** 提案，修复了 C++ 中一个长达 40 年的歧义问题：

- 在可变参数模板（variadic template）展开中，最后一个参数后的逗号现在有了明确的语义定义，类似英语中"牛津逗号"的规范地位
- 此改动将消除 Clang 与 GCC 在部分模板元编程场景下的编译器警告差异，使两者行为统一
- 对大多数应用层 C++ 开发者日常影响有限，但对 STL 和 Boost 等基础库维护者有直接意义

**为什么重要：** 标准化细节决定跨编译器兼容性上限，此改动为 C++26 的模板库生态铺平道路。
