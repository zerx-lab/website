### LiteLLM PyPI 供应链攻击：恶意包植入凭证窃取后门，72 分钟完成披露 ⭐⭐⭐⭐⭐

**核心要点：**

- **2026 年 3 月 24 日**，恶意版本 `litellm` v1.82.8 被上传至 PyPI，内含一个 `.pth` 文件（`litellm_init.pth`），该文件在任意 Python 启动时自动执行，无需任何用户操作
- 恶意代码通过 base64 编码的 `subprocess.Popen` 载荷启动子进程，目标是窃取 **SSH 密钥、AWS 密钥、GCP 凭证、Kubernetes token、`.env` 文件、数据库密码、加密钱包、Shell 历史记录**，加密后发往 `https://models.litellm.cloud/`
- 攻击者还尝试安装 **systemd 持久化服务** 并在集群内创建特权 Kubernetes Pod 以横向扩散；感染系统因自我复制触发 **1.1 万个子进程**，导致系统冻结——这一异常帮助工程师发现攻击

**技术解读：**

`.pth` 文件攻击向量是 Python 生态中最隐蔽的供应链攻击方式之一：Python 在启动时会自动执行 site-packages 目录下所有 `.pth` 文件中包含 `import` 或可执行代码的行，不需要任何直接导入。攻击者利用这一机制实现了"零感知持久执行"。LiteLLM 是目前最广泛使用的 LLM 代理/统一 API 库之一（PyPI 周下载量超百万），供应链攻击目标的选择精准打击了 AI 基础设施从业者。FutureSearch 工程师在发现异常后借助 **Claude Code** 完成了系统取证和根因分析，从症状发现到公开披露仅用了 **72 分钟**。

**背景上下文：**

随着 AI 工具链的 PyPI 包快速增殖，供应链安全已成为 AI 基础设施的核心风险。该事件与 2021 年 `ua-parser-js`、2022 年 PyTorch nightly 投毒事件一脉相承，但攻击目标从通用开发者转向了掌握 GPU 集群和云凭证的 AI 工程师——后者的受损成本更高。

**开发者行动建议：**

- **立即检查**：`pip show litellm` 确认版本，如为 v1.82.8 请立即卸载并轮换所有相关凭证
- 部署 `pip-audit` 或 `safety` 进行依赖漏洞扫描，并在 CI/CD 中固化依赖版本（使用 `requirements.txt` + hash 校验）
- 将 `~/.pth` 文件检测加入安全基线扫描：`python -c "import site; print(site.getsitepackages())"`

### Cloudflare Gen 13 服务器：用 Rust 重写换来 2 倍吞吐，50% 功耗优化 ⭐⭐⭐⭐⭐

**核心要点：**

- Cloudflare 将服务器从 AMD EPYC Genoa-X（96 核，每核 12MB L3 缓存）升级至 AMD EPYC Turin 9965（**192 核，每核仅 2MB L3 缓存**），核心数翻倍但单核缓存缩减 83%
- 在旧的 FL1 技术栈（NGINX + LuaJIT）下测试 Gen 13，因缓存减少导致高负载时延迟超标 **50%** 以上——单靠换硬件无法奏效
- 解决方案：同步上线全新 **FL2 请求处理层**（基于 Rust + Pingora + Oxy 框架），彻底消除对大缓存的依赖，最终实现：**吞吐量 ×2、机架吞吐提升 60%、单瓦性能提升 50%**

**技术解读：**

这是一个典型的"硬件-软件协同设计"案例。传统 NGINX/LuaJIT 架构依赖大 L3 缓存来掩盖代码热路径的低效——当缓存缩减后，性能悬崖暴露。Cloudflare 选择正面解决而非妥协：用 Rust 重写核心请求路径，消除了 GC 停顿和不必要的内存分配，使延迟可控地运行在更少的缓存上。这与 Cloudflare 此前将 CPU 密集型服务（如 TLS 握手）迁移至 Rust 的路径一脉相承，也是 Pingora 框架从 2022 年开源以来最具影响力的生产验证。

**背景上下文：**

Cloudflare 目前在 300 个城市运营超过 330 个数据中心，边缘性能直接影响数以亿计用户的网络体验。Gen 13 的上线将使 Cloudflare 在不提高功耗预算的前提下，将全球处理能力翻倍——这为其 AI 推理、R2 存储和 Workers 平台的下一轮扩张打下基础。

**开发者行动建议：**

- Rust 性能工程师：阅读 Cloudflare 博客了解 FL2 的架构决策，Pingora 和 Oxy 框架均已开源
- 基础设施架构师：Gen 13 案例是"先测 profile 再换硬件"的教科书示范——新硬件未必直接提升性能
- 关注 Cloudflare 后续关于 Workers AI 在 Gen 13 上的性能公告

**相关链接：**

### OpenTelemetry Profiles 进入公开 Alpha：可观测性的第四信号标准化 ⭐⭐⭐⭐

OpenTelemetry Profiling SIG 宣布 **Profiles 进入公开 Alpha**，这是继 Traces、Metrics、Logs 之后，OTel 将持续性生产 Profiling 纳入统一可观测性标准的重要里程碑。

主要内容：
- **统一数据格式**：基于去重栈表示和字典表的高效编码，可与 pprof 格式**无损双向转换**，兼容现有工具链
- **eBPF Profiling Agent**（基于 Elastic 捐赠代码）：全系统 Linux 无侵入式采集，本次新增 ARM64 对 Node.js 的支持、Erlang/Elixir 的 BEAM 支持，以及 Ruby 和 .NET 的兼容性改进
- **OTel Collector 集成**：新增 Kubernetes 元数据富化处理器，可与 Traces 关联分析

**为什么重要：** 生产环境 Profiling 长期缺乏统一标准，开发者需要学习多套互不兼容的工具（pprof、pyspy、async-profiler 等）。OTel Profiles 的标准化将使"从 trace 直接跳转到相关 profile"成为开箱即用的体验，大幅降低性能调优的认知成本。
